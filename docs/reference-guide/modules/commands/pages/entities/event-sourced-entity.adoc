= Event-Sourced Entities
:navtitle: Event-sourced entities

An event-sourced entity stores no state directly.
Instead, every state change is recorded as an event, and the entity's current state is reconstructed by replaying those events in order.
This makes the event log the single source of truth, giving you a complete and immutable audit trail of everything that ever happened to the entity.

[TIP]
====
If you are unfamiliar with event sourcing, see xref:events:index.adoc[Events] for an introduction to the concept.
====

== Defining an event-sourced entity

An event-sourced entity consists of three types of methods:

* **Creational command handlers**: static methods that handle the command that creates the entity.
They validate the command and append the creation event.
* **Instance command handlers**: instance methods that handle commands on an already-existing entity.
They validate the command against current state and append events.
* **Event sourcing handlers**: private instance methods that apply events to update the entity's state fields.
They must never contain business logic, only state assignments.

The `@EventSourcedEntity` annotation (or `@EventSourced` in Spring environments) marks the class so Axon can discover and register it.

[tabs]
====
Spring Boot::
+
[source,java]
----
import org.axonframework.eventsourcing.annotation.EventSourcingHandler;
import org.axonframework.eventsourcing.annotation.reflection.EntityCreator;
import org.axonframework.extension.spring.stereotype.EventSourced;
import org.axonframework.messaging.commandhandling.annotation.CommandHandler;
import org.axonframework.messaging.eventhandling.gateway.EventAppender;

@EventSourced(tagKey = "courseId") // <1>
public class CourseEntity {

    private String courseId;
    private String title;
    private int capacity;
    private int enrolledCount;

    @CommandHandler // <2>
    public static String handle(CreateCourseCommand cmd, EventAppender eventAppender) {
        if (cmd.capacity() <= 0) {
            throw new IllegalArgumentException("Capacity must be positive");
        }
        eventAppender.append(new CourseCreatedEvent(cmd.courseId(), cmd.title(), cmd.capacity())); // <3>
        return cmd.courseId(); // <4>
    }

    @CommandHandler
    public void handle(EnrollStudentCommand cmd, EventAppender eventAppender) {
        if (enrolledCount >= capacity) { // <5>
            throw new IllegalStateException("Course is full");
        }
        eventAppender.append(new StudentEnrolledEvent(courseId, cmd.studentId()));
    }

    @EventSourcingHandler // <6>
    private void on(CourseCreatedEvent event) {
        this.courseId = event.courseId();
        this.title = event.title();
        this.capacity = event.capacity();
    }

    @EventSourcingHandler
    private void on(StudentEnrolledEvent event) {
        this.enrolledCount++;
    }

    @EntityCreator // <7>
    protected CourseEntity() {
    }
}
----
+
<1> `tagKey = "courseId"` tells Axon which tag to filter events by when loading this entity's stream.
If omitted, Axon falls back to the entity's simple class name.
+
<2> A static method with `@CommandHandler` is the creational handler.
It must be static because no entity instance exists yet when the creation command arrives.
+
<3> Events are appended via `EventAppender`, which is injected as a parameter.
Never modify state directly in a command handler; always go through events.
+
<4> Return the entity identifier so the caller knows which entity was created.
+
<5> Instance command handlers read current state to make decisions.
This state was built up by replaying past events via the `@EventSourcingHandler` methods.
+
<6> Event sourcing handlers update state.
They are called when replaying historical events to reconstruct the entity, and again for newly appended events.
+
<7> The `@EntityCreator` constructor tells Axon how to create an empty entity instance before applying events.
See xref:commands:command-handlers.adoc#entity-creation-with-entitycreator[Entity creation with `@EntityCreator`] for all three creation patterns.

Plain Java::
+
[source,java]
----
import org.axonframework.eventsourcing.annotation.EventSourcedEntity;
import org.axonframework.eventsourcing.annotation.EventSourcingHandler;
import org.axonframework.eventsourcing.annotation.reflection.EntityCreator;
import org.axonframework.messaging.commandhandling.annotation.CommandHandler;
import org.axonframework.messaging.eventhandling.gateway.EventAppender;

@EventSourcedEntity(tagKey = "courseId") // <1>
public class CourseEntity {

    private String courseId;
    private String title;
    private int capacity;
    private int enrolledCount;

    @CommandHandler
    public static String handle(CreateCourseCommand cmd, EventAppender eventAppender) {
        if (cmd.capacity() <= 0) {
            throw new IllegalArgumentException("Capacity must be positive");
        }
        eventAppender.append(new CourseCreatedEvent(cmd.courseId(), cmd.title(), cmd.capacity()));
        return cmd.courseId();
    }

    @CommandHandler
    public void handle(EnrollStudentCommand cmd, EventAppender eventAppender) {
        if (enrolledCount >= capacity) {
            throw new IllegalStateException("Course is full");
        }
        eventAppender.append(new StudentEnrolledEvent(courseId, cmd.studentId()));
    }

    @EventSourcingHandler
    private void on(CourseCreatedEvent event) {
        this.courseId = event.courseId();
        this.title = event.title();
        this.capacity = event.capacity();
    }

    private void on(StudentEnrolledEvent event) {

    }

    protected CourseEntity() {
    }
}
----
+
<1> Use `@EventSourcedEntity` (non-Spring) instead of `@EventSourced`.
Subtypes do not need a separate annotation.
====

== Event stream filtering and command routing

`tagKey` on `@EventSourcedEntity` tells Axon which tag key to use when filtering the event store for this entity's events.
The events themselves must carry a matching tag, added with `@EventTag`:

[source,java]
----
import org.axonframework.eventsourcing.annotation.EventTag;
import org.axonframework.extension.spring.stereotype.EventSourced;

public record CourseCreatedEvent(
    @EventTag String courseId, // <1>
    String title,
    int capacity
) {}

@EventSourced(tagKey = "courseId") // <2>
public class CourseEntity {
    // ...
}
----

<1> `@EventTag` creates a tag from the annotated field.
By default the tag key is the field name, so `@EventTag String courseId` produces `Tag("courseId", value)`.
<2> `tagKey = "courseId"` tells Axon to load this entity's events by filtering on tag key `"courseId"`, matching the tag produced by `@EventTag`.

If `tagKey` is omitted on the entity, Axon falls back to the entity's simple class nameŌĆöset it explicitly to keep the tag key stable across refactoring.

For instance command handlers, Axon also needs to know which entity instance to load.
Annotate the identifier field on the command with `@TargetEntityId`:

[source,java]
----
import org.axonframework.modelling.annotation.TargetEntityId;

public record EnrollStudentCommand(
    @TargetEntityId String courseId, // <1>
    String studentId
) {}
----

<1> Axon reads this field to load the correct entity instance from the event store before invoking the handler.
Not required on the static creational command handler, since no entity exists yet at that point.

[NOTE]
====
`@Command(routingKey = "...")` is a separate concern: it sets the routing key used by a distributed command bus (such as Axon Server) to route the command to the right application node.
It does not affect which entity instance is loaded.
====

== Separation of concerns

Command handlers validate and publish events; they must never write to state fields.
Event sourcing handlers update state; they must contain no business logic.

This separation matters because event sourcing handlers also run during replay when the entity is loaded from its event streamŌĆöany business logic there would execute unexpectedly.

[TIP]
====
The xref:testing:index.adoc[Test Fixture] catches state mutations in command handlers automatically.
====

== Appending events with metadata

Pass a `Metadata` instance as the second argument to attach metadata (correlation IDs, user context, etc.) to an event:

[source,java]
----
import org.axonframework.messaging.core.Metadata;

@CommandHandler
public void handle(EnrollStudentCommand cmd, EventAppender eventAppender) {
    Metadata metadata = Metadata.from(Map.of("enrolledBy", cmd.operatorId()));
    eventAppender.append(new StudentEnrolledEvent(courseId, cmd.studentId()), metadata);
}
----

To attach the same metadata to multiple events: `eventAppender.append(List.of(event1, event2), metadata)`.

== Configuring event-sourced entities

[tabs]
====
Spring Boot::
+
Annotate the entity with `@EventSourced`.
Axon's auto-configuration detects and registers it automatically.
+
[source,java]
----
@EventSourced
public class CourseEntity {
    // ...
}
----

Plain Java::
+
Register the entity using `EventSourcingConfigurer`:
+
[source,java]
----
import org.axonframework.eventsourcing.configuration.EventSourcedEntityModule;
import org.axonframework.eventsourcing.configuration.EventSourcingConfigurer;

EventSourcingConfigurer configurer = EventSourcingConfigurer.create();
configurer.registerEntity(
    EventSourcedEntityModule.autodetected(String.class, CourseEntity.class) // <1>
);
----
+
<1> The first argument is the identifier type; the second is the entity class.
`autodetected` reads annotations from the class automatically.
====

== Snapshots

Snapshots capture entity state at a point in time so Axon only needs to replay events after the snapshot, avoiding the cost of replaying the full event stream.

[NOTE]
====
Snapshot policy is only configurable through the _declarative_ builder (`EventSourcedEntityModule.declarative`).
The annotation-based `autodetected` helper does not expose a snapshot configuration point.
If you need snapshots, build the module with `declarative` instead of `autodetected`, providing the messaging model, entity factory, and criteria resolver explicitly.
====

[tabs]
====
Spring Boot::
+
Declare the module as a Spring bean using the declarative builder.
Do not also annotate the entity class with `@EventSourced`, as that would register the entity twice.
Also declare a `SnapshotStore` beanŌĆöAxon does not auto-configure one.
+
[source,java]
----
import org.axonframework.eventsourcing.EventSourcedEntityFactory;
import org.axonframework.eventsourcing.annotation.AnnotationBasedEventCriteriaResolver;
import org.axonframework.eventsourcing.configuration.EventSourcedEntityModule;
import org.axonframework.eventsourcing.snapshot.api.SnapshotPolicy;
import org.axonframework.eventsourcing.snapshot.inmemory.InMemorySnapshotStore;
import org.axonframework.eventsourcing.snapshot.store.SnapshotStore;
import org.axonframework.messaging.core.MessageTypeResolver;
import org.axonframework.messaging.core.annotation.ParameterResolverFactory;
import org.axonframework.messaging.core.conversion.MessageConverter;
import org.axonframework.messaging.eventhandling.conversion.EventConverter;
import org.axonframework.modelling.annotation.AnnotationBasedEntityIdResolver;
import org.axonframework.modelling.entity.annotation.AnnotatedEntityMetamodel;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class CourseEntityConfiguration {

    @Bean
    public SnapshotStore snapshotStore() {
        return new InMemorySnapshotStore(); // <1>
    }

    @Bean
    public EventSourcedEntityModule<String, CourseEntity> courseEntityModule() {
        return EventSourcedEntityModule.declarative(String.class, CourseEntity.class)
            .messagingModel((config, builder) -> AnnotatedEntityMetamodel.forConcreteType( // <2>
                CourseEntity.class,
                config.getComponent(ParameterResolverFactory.class),
                config.getComponent(MessageTypeResolver.class),
                config.getComponent(MessageConverter.class),
                config.getComponent(EventConverter.class)
            ))
            .entityFactory(c -> EventSourcedEntityFactory.fromNoArgument(CourseEntity::new)) // <3>
            .criteriaResolver(c -> new AnnotationBasedEventCriteriaResolver<>( // <4>
                CourseEntity.class, String.class, c
            ))
            .entityIdResolver(c -> new AnnotationBasedEntityIdResolver<>()) // <5>
            .snapshotPolicy(c -> SnapshotPolicy.afterEvents(50)) // <6>
            .build();
    }
}
----
+
<1> `InMemorySnapshotStore` stores snapshots in memoryŌĆöfor development and testing only.
In production, replace this with a persistent `SnapshotStore` implementation backed by a database or the event store.
+
<2> Builds the entity metamodel from annotations on `CourseEntity` (`@CommandHandler`, `@EventSourcingHandler`).
+
<3> Creates a new entity instance via the `@EntityCreator` no-arg constructor.
+
<4> Reads `@EventSourcedEntity(tagKey = ...)` from `CourseEntity` to build the `EventCriteria` for event stream filtering.
+
<5> Reads `@TargetEntityId` from command payloads to identify which entity instance to load.
+
<6> `SnapshotPolicy.afterEvents(50)` triggers a snapshot after more than 50 events have been applied since the last snapshot.

Plain Java::
+
[source,java]
----
import org.axonframework.eventsourcing.EventSourcedEntityFactory;
import org.axonframework.eventsourcing.annotation.AnnotationBasedEventCriteriaResolver;
import org.axonframework.eventsourcing.configuration.EventSourcedEntityModule;
import org.axonframework.eventsourcing.configuration.EventSourcingConfigurer;
import org.axonframework.eventsourcing.snapshot.api.SnapshotPolicy;
import org.axonframework.eventsourcing.snapshot.inmemory.InMemorySnapshotStore;
import org.axonframework.eventsourcing.snapshot.store.SnapshotStore;
import org.axonframework.messaging.core.MessageTypeResolver;
import org.axonframework.messaging.core.annotation.ParameterResolverFactory;
import org.axonframework.messaging.core.conversion.MessageConverter;
import org.axonframework.messaging.eventhandling.conversion.EventConverter;
import org.axonframework.modelling.annotation.AnnotationBasedEntityIdResolver;
import org.axonframework.modelling.entity.annotation.AnnotatedEntityMetamodel;

EventSourcingConfigurer configurer = EventSourcingConfigurer.create();

configurer.componentRegistry(r -> r.registerComponent(
    SnapshotStore.class, c -> new InMemorySnapshotStore() // <1>
));

configurer.registerEntity(
    EventSourcedEntityModule.declarative(String.class, CourseEntity.class)
        .messagingModel((config, builder) -> AnnotatedEntityMetamodel.forConcreteType( // <2>
            CourseEntity.class,
            config.getComponent(ParameterResolverFactory.class),
            config.getComponent(MessageTypeResolver.class),
            config.getComponent(MessageConverter.class),
            config.getComponent(EventConverter.class)
        ))
        .entityFactory(c -> EventSourcedEntityFactory.fromNoArgument(CourseEntity::new)) // <3>
        .criteriaResolver(c -> new AnnotationBasedEventCriteriaResolver<>( // <4>
            CourseEntity.class, String.class, c
        ))
        .entityIdResolver(c -> new AnnotationBasedEntityIdResolver<>()) // <5>
        .snapshotPolicy(c -> SnapshotPolicy.afterEvents(50)) // <6>
        .build()
);
----
+
<1> `InMemorySnapshotStore` stores snapshots in memoryŌĆöfor development and testing only.
In production, replace this with a persistent `SnapshotStore` implementation backed by a database or the event store.
+
<2> Builds the entity metamodel from annotations on `CourseEntity` (`@CommandHandler`, `@EventSourcingHandler`).
+
<3> Creates a new entity instance via the `@EntityCreator` no-arg constructor.
+
<4> Reads `@EventSourcedEntity(tagKey = ...)` from `CourseEntity` to build the `EventCriteria` for event stream filtering.
+
<5> Reads `@TargetEntityId` from command payloads to identify which entity instance to load.
+
<6> `SnapshotPolicy.afterEvents(50)` triggers a snapshot after more than 50 events have been applied since the last snapshot.
====
